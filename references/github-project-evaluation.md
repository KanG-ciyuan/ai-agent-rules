# GitHub 开源项目评估指南

> 动手三问第②条（值不值得）的落地工具。决定是否采用一个 GitHub 开源项目时，按五个维度做评估，不凭感觉判断。

## 五维度评估

### 1. 许可证

- 看 `LICENSE` 文件，确认是否开源、能否商用
- MIT / Apache 2.0 / BSD → 宽松，自由使用
- GPL → 传染性，用了它你的项目也必须开源
- 无许可证 → 默认保留全部权利，不能合法使用

### 2. 体积

- 克隆前看仓库大小（GitHub 页面右侧 About 区域）
- 算上依赖后的真实占用（`brew deps --tree` 或 `npm ls`）
- 判断：体积是否匹配功能？有没有不必要的依赖？

### 3. 语言栈

- 用什么语言写的？和你的技术栈兼容吗？
- 出问题时你自己能改吗？
- 如果语言不熟 → 学习成本算进"消耗"

### 4. 网络依赖

- 安装时需要从哪下载？GitHub？npm？PyPI？
- 国内网络能否直接访问？需要代理吗？
- 运行时需要联网吗？数据上传到外部吗？

### 5. 功能重叠

- 已有工具/系统有没有类似功能？
- 新工具带来的是增量价值，还是功能重叠维护成本？
- 如果只是替换 → 替换的收益大于迁移成本吗？

## 评估输出格式

```
📊 项目评估：[项目名]

- 许可证：[类型，是否可用]
- 体积：[仓库大小 + 依赖估算]
- 语言栈：[语言/框架，你的熟悉度]
- 网络：[安装源 / 运行联网需求 / 代理是否必要]
- 功能重叠：[与现有工具的关系]

结论：[建议采用 / 谨慎 / 不建议]
```

## 经验法则

- 五个维度都绿灯 → 可以装
- 有一个红灯 → 告诉用户风险，等确认
- 有两个以上红灯 → 主动建议替代方案
